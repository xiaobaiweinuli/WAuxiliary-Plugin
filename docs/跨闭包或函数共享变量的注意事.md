跨闭包/函数共享变量的注意事项

在某些脚本引擎（如动态加载的插件环境）中，当你在一个函数中修改了某个顶层变量，但在另一个闭包、回调或函数中读取它时，可能会发现读取到的仍是旧值或 null。这是因为引擎在实现闭包时，可能将变量以副本或不同作用域的形式捕获，导致看起来是同一个变量，实际上读写的是两份不同的数据。

典型现象：
你定义了一个全局变量 mainWindow，在 createWindow() 方法中给它赋值为 newWindow，但在点击事件的回调中访问 mainWindow 时却是 null，即使 createWindow() 已经成功执行过。

解决方案：
使用一个全局共享的容器（如 HashMap）来存储这些需要跨多个函数/闭包访问的数据。所有地方都通过容器的 get/put 方法操作同一份数据，不再依赖脚本引擎的变量绑定机制。这样就能保证在任何上下文中拿到的都是同一个对象引用。

示例：

```java
// 原来使用顶层变量可能出问题
AlertDialog myDialog = null;

void createDialog() {
    myDialog = new AlertDialog(); // 赋值
}

void onClick() {
    // 这里的 myDialog 可能仍然是 null
    if (myDialog != null) {
        myDialog.show();
    }
}

// 改为使用 HashMap 存储后
HashMap globalStore = new HashMap();

void createDialog() {
    AlertDialog dlg = new AlertDialog();
    globalStore.put("myDialog", dlg);
}

void onClick() {
    AlertDialog dlg = (AlertDialog) globalStore.get("myDialog");
    if (dlg != null) {
        dlg.show();
    }
}
```

优点：

· 彻底避免闭包引用不同步的坑。
· 代码清晰，全局共享数据集中管理。
· 无论函数调用栈多么复杂，都能保证数据一致性。

适用场景：

· 需要在不同回调、线程、闭包间共享的 UI 对象（如对话框、窗口）。
· 多个方法需要协作读写同一份状态数据时。
· 脚本引擎的变量作用域行为不确定时，作为一种防御性编程策略。

记住这条经验：在不可靠的闭包环境中，用容器代替直接变量传递，让所有访问都指向同一个物理存储。
